# ADX External Tables

An ADX external table represents an entity stored within a shared data lake.  
Entity paths and partition structures are designed around dominant query constraints and archival requirements.  Entities are grouped into containers inside a single storage account and single ADX database.  Multiple containers gives:
- clean organization
- low complexity
- easy querying
- future flexibility

without introducing namespace-routing.  Having seperare container even has infrastructure benefits:
- migrate telemetry separately
- replicate analytics elsewhere
- isolate costs
- split workloads

without redesigning paths.  ADX must have access to every container.

Query performance is the primary design concern. For this reason, information is stored using the Parquet file [format](https://parquet.apache.org/docs/) rather than JSON or CSV.  Performance depends on:
- partition pruning
- parquet layout
- file counts
- file sizes
- storage latency

ADX tables names must be unique to the ADX database regardless storage settings like distinct storage containers.  ADX table and column names should follow established forestry and logistics standards where possible, including:
- PapiNet
- StanForD
- third-party extensions such as Biometria delivery source definitions

Kusto external tables support composition and reuse through functions and templates, but do not support inheritance, abstract base tables, or enumeration types. Schemas are flat and independent objects.

Partition columns are derived from storage paths rather than blob content.  Partitioning is path-based inside the external table, independent of the container.  Partition dimensions should therefore reflect the most common query predicates.

Date dimensions within paths and partitions also support lifecycle operations such as:
- archival
- retention
- blob tier transitions

Most operative telemetry volumes are relatively low. As a result, day-level partitioning is unnecessary and can increase partition fragmentation.

Using static path segments is preferred over filename prefixes because path-based filtering improves query pruning and long-term storage organization.

## Business

Contracts in Sweden are either:
- original agreements
- trading agreements

between buyer and seller parties for forest products.

Contract references bind operative activities such as production and logistics to business configurations including agreement type, delivery terms, and ownership.

Because nearly all operative events reference a contract, contract identifiers are strong partition candidates.

A contract reference also contains a starting date suitable for archival-oriented partitioning.

## Operative

Operative information consists primarily of machine-generated telemetry from sources such as:
- harvesters
- forwarders
- load weighing systems
- quality measurement systems

The data lake organizes telemetry using the following path structure:

```kusto
pathformat=(
    "telemetry"
    "/source=" source
    "/metric=" metric
    "/contract=" contract
    "/year=" year
    "/month=" month
)
```

with partition dimensions:

```kusto
partition by (
    source:string,
    metric:string,
    contract:string,
    year:int,
    month:int
)
```

The `source` dimension represents the originating telemetry domain, while `metric` represents the measured entity or aggregation category.

Examples:
- source=harvesting
- source=forwarding
- metric=stem
- metric=log
- metric=load

Telemetries are static and not versioned.  They are small parquet blobs never consolidated into a larger query-friendly blob.

## Analysis

Operative telemetries are analysized then deemed ready for user querying meaning the parquet blobs will contain multiple analysized telemetries aiming for 100-500 MB in size.  Larger parquet blobs:
- reduce metadata overhead
- reduce cloud object listing overhead
- improve scan efficiency
- improve compression ratios
- reduce open/close operations

Parquet consolidation occurs when the number of blobs tips over a threadhold of total bytes in a directory.

## Production

Machines operating in Sweden commonly use the StanForD Classic and StanForD 2010 formats.

StanForD references:
- [StanForD 2010](https://stanford2010.skogforsk.se/version-4-0/index-4-0.html)

Production telemetry originates from machine systems responsible for:
- harvesting
- forwarding
- quality measurement

Telemetry metrics include:
- stems
- logs
- loads
- production aggregates
- quality aggregates

### Harvesting

Harvesting telemetry primarily represents stem- and log-level production generated by harvester systems.

### Forwarding

Forwarding telemetry represents transport, loading, unloading, and load-balancing activities generated by forwarder systems.

### Quality

Quality telemetry represents measurements and classifications related to timber grading, dimensions, defects, moisture, and delivery validation.
